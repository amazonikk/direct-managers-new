# Direct Managers Dashboard

Автоматичний GitHub Pages дашборд для двох direct-менеджерів:

- **Тая:** https://docs.google.com/spreadsheets/d/1kjC6t8IMlipgE827-yqyJWUK0kZEU01gYxrNPn1sKSE/edit
- **Катерина:** https://docs.google.com/spreadsheets/d/1nLX2GGih9k6UJHDsH33_qK6IGquxzLGvWW7FQfMzt9Q/edit

Дашборд показує по місяцях і календарних тижнях:

- загальні чати;
- отримані номери;
- загальну конверсію;
- окремо Таю і Катерину;
- окремо кожен акаунт;
- загальне порівняння двох direct-менеджерів.

## Як це працює

1. GitHub Actions кожні **15 хвилин** завантажує обидві Google Таблиці у форматі XLSX.
2. Скрипт `scripts/build-data.mjs` автоматично знаходить аркуші, де є одночасно:
   - дата;
   - загальні чати або складові чатів;
   - загальна кількість номерів.
3. Скрипт створює агрегований файл `data/report.json`.
4. GitHub Pages публікує оновлений дашборд.

У сайт не копіюються коментарі, посилання, листи чи інші внутрішні тексти. У JSON потрапляють лише:

- дата;
- кількість чатів;
- кількість номерів;
- назва аркуша/акаунта.

## Запуск через GitHub

### 1. Створіть репозиторій

Створіть новий репозиторій, наприклад:

`direct-managers-dashboard`

### 2. Завантажте файли

Завантажте в корінь репозиторію **весь вміст цієї папки**, включно з прихованою папкою:

`.github/workflows/`

Структура має виглядати так:

```text
.github/
  workflows/
    deploy-pages.yml
data/
  report.json
scripts/
  build-data.mjs
.nojekyll
app.js
index.html
package.json
README.md
styles.css
```

### 3. Увімкніть GitHub Pages

У репозиторії відкрийте:

`Settings → Pages → Build and deployment → Source → GitHub Actions`

### 4. Запустіть перше оновлення

Відкрийте:

`Actions → Update data and deploy Pages → Run workflow`

Після успішного запуску посилання на сайт буде видно в:

`Settings → Pages`

Надалі GitHub автоматично оновлюватиме дані кожні 15 хвилин.

## Важлива умова для Google Таблиць

Обидві таблиці повинні мати доступ:

`Усі, хто має посилання → Читач`

Інакше GitHub Actions не зможе завантажити XLSX-файл.


## Автоматичне додавання нових місяців

Новий місяць не потрібно додавати в код вручну. Якщо в будь-якій підключеній таблиці з’являться рядки з датами серпня, вересня або іншого місяця, після наступного запуску GitHub Actions місяць автоматично з’явиться у перемикачі. Майбутні дати також враховуються.

Картки тижнів клікабельні. Натискання відкриває детальний перегляд тижня з розбивкою по днях, direct-менеджерах і акаунтах.

## Ручне оновлення

У будь-який момент можна примусово оновити дані:

`Actions → Update data and deploy Pages → Run workflow`

Кнопка **«Оновити зараз»** у самому дашборді повторно завантажує останню версію, яку вже опублікував GitHub Pages.

## Локальний запуск

Потрібен Node.js 20+.

```bash
npm install
npm run build:data
python -m http.server 8080
```

Після цього відкрийте:

`http://localhost:8080`

Не відкривайте `index.html` подвійним кліком через `file://`, тому що браузер може заблокувати завантаження `data/report.json`.

## Як додати або замінити таблицю

Відкрийте файл:

`scripts/build-data.mjs`

На початку файлу є масив `MANAGERS`:

```js
const MANAGERS = [
  {
    id: "taya",
    name: "Тая",
    spreadsheetId: "ID_ТАБЛИЦІ"
  }
];
```

Замініть тільки `spreadsheetId` або додайте нового direct-менеджера за аналогією.

## Які аркуші враховуються

Автоматично враховуються аркуші зі структурою на кшталт:

- `ДАТА`;
- `ЗАГАЛЬНІ ЧАТИ`;
- `ВСЬОГО НОМЕРІВ`.

Для Telegram також підтримується сума колонок:

- `к-сть чатів`;
- `к-сть звернень`;
- `к-сть написаних`;
- `Всього номерів разом`.

Аркуші на кшталт `ФБ-Пости`, `Вакансії`, `Пошта`, `ЮТУБ`, де немає одночасно чатів і номерів, пропускаються автоматично.

## Якщо змінилася структура таблиці

Якщо колонки перейменували, перевірте запуск у вкладці `Actions`. У помилці буде вказано, чи не вдалося завантажити таблицю. Список пропущених аркушів також записується в `data/report.json` у полі `ignoredSheets`.
